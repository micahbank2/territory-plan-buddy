# UI Review — Phases 5-9 Visual Surfaces

## TL;DR

The five surfaces shipped this session are visually consistent with the rest of the app, faithful to the existing token system, and demonstrate disciplined reuse of the `glass-card`/`bg-muted/30`/`bg-card` vocabulary established earlier. Aggregate quality is high (**19/24** averaged across surfaces) but there are several recurring accessibility and hierarchy gaps — most notably the lack of `aria-label` on the segmented stage bar segments (only `aria-label` on `<div>` is non-semantic), a print export that hides ALL `<button>` elements (which removes the print button before the user can press it on the rendered page), and pervasive use of `text-[10px]` / `text-[11px]` arbitrary type sizes that drift below WCAG-recommended minimums and are not in the type scale.

## Findings (ordered by severity)

### CRITICAL (must fix before next ship)

**C1. Print stylesheet hides the trigger that initiates print** — `src/index.css:365-367`

```css
button {
  display: none !important;
}
```

This rule is correct for the *printed page* but the print preview generated by `window.print()` from a button means *if the user clicks a "Print" button on the page itself*, the preview will not show that button (good) but more critically: the rule applies the moment the user opens the browser print menu (Cmd+P), and any contextual buttons that should remain — e.g. external "Print" links inside printable content — vanish. More importantly, the rule has a usability footgun: there is no way for a user to cancel a hidden interaction. **Fix:** scope the rule to `[data-no-print] button, .sticky button { display: none !important; }`, or add `button` to a `data-no-print` parent wrapping the header.

### HIGH (fix this week)

**H1. PipelineForecastBar segments have `aria-label` on a non-interactive `<div>`** — `src/components/PipelineForecastBar.tsx:114-118`

```tsx
<div
  className={cn("h-full transition-all duration-500", ...)}
  style={{ width: `${widthPct}%` }}
  aria-label={`${b.stage}: ${b.count} deals, $${b.weighted.toLocaleString()} weighted`}
/>
```

Screen readers ignore `aria-label` on a plain `<div>` with no `role`. The tooltip is keyboard-inaccessible because the trigger is a non-focusable div. **Fix:** add `role="img"` (the segments are essentially mini bar-chart icons) and/or wrap the segment in a `<button type="button">` so keyboard tab + tooltip activation works. The shadcn `Tooltip` requires a focusable child for keyboard reveal — this is currently mouse-only.

**H2. MeetingPrepDialog has no `DialogDescription` and no live region for the loading state** — `src/components/MeetingPrepDialog.tsx:131-143`

The `<DialogContent>` ships a `<DialogTitle>` but no `<DialogDescription>` (Radix logs a warning in dev). The "Generating meeting prep..." block (line 142) is not in an `aria-live="polite"` region, so screen readers announce the dialog opening but never hear that work is in progress. **Fix:** add `<DialogDescription className="sr-only">Generates a one-page meeting brief from prospect history.</DialogDescription>` and wrap the loading block in `<div role="status" aria-live="polite">`.

**H3. Hardcoded amber/emerald color classes proliferate without dark-mode review** — multiple files

77 occurrences of `amber-*` / `emerald-*` / `slate-*` / `violet-*` / `indigo-*` / `orange-*` / `blue-*` Tailwind palette classes across 17 files. Phase 5-9 contributions include `bg-amber-500/15`, `text-amber-600`, `text-emerald-500`, `bg-emerald-500/5` in `RecommendationCard.tsx:18-19`, `PipelineForecastBar.tsx:57-59`, `TodayPage.tsx:77,166,202`, and `forecast.ts:24-32`. These bypass the `--success` / `--warning` / `--destructive` semantic tokens defined in `index.css:25-28` (light) and `:60-63` (dark) and ship the same hue regardless of theme. **Fix:** route all severity color through `text-success`, `text-warning`, `text-destructive` token classes (or add `--info` if a fourth severity is needed). The `STAGE_BAR_COLORS` map in `forecast.ts:24-33` is the only place where palette-direct colors are appropriate (categorical encoding) and should stay.

### MEDIUM (queue for next polish pass)

**M1. Pervasive `text-[10px]` and `text-[11px]` arbitrary sizes** — 28 occurrences across `RecommendationCard.tsx`, `PipelineForecastBar.tsx`, `TodayPage.tsx`, plus 21 pre-existing.

Specifically in this session: `RecommendationCard.tsx:26,48,52,67`, `PipelineForecastBar.tsx:75,91,106,123,131-132`, `TodayPage.tsx:271`. 10px is below the 12px minimum recommended for body type and below the 11px Apple HIG minimum. These also fall outside the Tailwind type scale (`text-xs` = 12px is the smallest token). **Fix:** standardize on `text-xs` (12px) for all label-style copy; reserve `text-[10px]` for genuine micro-labels and add a project utility class so it's intentional, not ambient.

**M2. PipelineForecastBar headline density saturates on mobile** — `src/components/PipelineForecastBar.tsx:64-93`

The headline row uses `flex items-center gap-6 flex-wrap` with up to four columns (Weighted / Raw Open / Booked / % Quota) plus the booked column (conditional). At <640px each item wraps to its own row but the `ml-auto text-right` on the % Quota column pulls it right while sibling columns are left-aligned, producing a visually-broken zig-zag. **Fix:** at the `flex-wrap` breakpoint, drop `ml-auto` (or replace the desktop dividers `w-px h-10 bg-border hidden sm:block` with a `grid grid-cols-2 sm:flex` layout so mobile reflows to a clean 2x2 grid).

**M3. RecommendationCard CalloutChip uses `border` + tinted bg + tinted text — borderline contrast on light theme** — `src/components/RecommendationCard.tsx:16-20`

`bg-amber-500/15 text-amber-700` on light yields ~3.8:1 contrast (WCAG-AA fails for body text under 14px; passes for "large text" at 18px+). The chip text is `text-[11px]` (line 26), which is decisively under-size. **Fix:** bump chip text to `text-xs` (12px) AND bold (`font-bold` instead of `font-semibold`) — already half-applied — to push past AA. Or change to `text-amber-800 dark:text-amber-300`.

**M4. MeetingPrepDialog PDF export uses inline hex hardcodes** — `src/components/MeetingPrepDialog.tsx:107-112`

```css
.header { border-bottom: 2px solid #2563eb; ... color: #111; }
.meta { color: #666; }
.footer { color: #999; }
```

This is a print-only document so the absence of theme tokens is defensible (the user is printing, not viewing in dark mode), but `#2563eb` is a tailwind blue-600, not the app primary `hsl(236 64% 57%)` which is roughly `#5158d3`. The exported PDF cosmetically diverges from the app brand. **Fix:** swap `#2563eb` → `#5158d3` (or compute from the CSS variable at export time).

**M5. TodayPage hero stat label `text-[10px]`** — `src/pages/TodayPage.tsx:271`

Same `text-[10px]` issue as M1, but worth calling out explicitly: the four hero metrics are the single most-scanned element on the daily briefing. Their *labels* render below recommended minimums. The values (`text-2xl font-black`) are fine. **Fix:** `text-[10px]` → `text-xs`.

**M6. LogActivityWidget submit copy length-shifts** — `src/components/LogActivityWidget.tsx:207`

```tsx
Log Activity{showFollowUp && taskText.trim() ? " + Create Task" : ""}
```

The button text grows from "Log Activity" → "Log Activity + Create Task" mid-form. On mobile, this re-flows the form below it once the user types into the task field. **Fix:** stabilize layout with `min-w-[180px]` on the button or fix the copy as "Log Activity & Follow-up" once the toggle is on (no per-keystroke change).

### LOW (nice-to-have)

**L1. RecommendationCard `style={{ color: rec.scoreColor }}`** — `src/components/RecommendationCard.tsx:53`

Inline style escapes Tailwind's theme. The `scoreColor` originates in `getScoreLabel` which returns hardcoded values. Acceptable but worth migrating to a token so dark-mode contrast stays correct. (Existing pattern repeated from `ScoreBadge.tsx:24` — not a Phase 6 regression.)

**L2. PipelineForecastBar empty state could be friendlier** — `src/components/PipelineForecastBar.tsx:46-53`

"No active pipeline · Add a deal to see your weighted forecast." is functional. A single-click action ("Add Deal" CTA in the empty card) would improve the empty-to-first-deal latency, but this is feature-creep, not a fix.

**L3. MeetingPrepDialog "Generating meeting prep..." copy could include prospect name** — `MeetingPrepDialog.tsx:142`

Minor: "Generating meeting prep for Shake Shack..." gives the user something to read while waiting on the LLM round-trip. Currently it's anonymous.

**L4. TodayPage section reveal order is hard-coded; section weights assume Hot prospects exist** — `TodayPage.tsx:84-228`

When a user has zero Hot prospects but plenty of overdue tasks, "Today's Plan" disappears and the page leads with "Overdue Tasks." That's correct, but the ranking logic isn't documented. Worth adding a comment block at line 53 spelling out the visual priority order. (Not a UI fix; a maintainability nit.)

**L5. LogActivityWidget — date popover `z-[60]` arbitrary z-index** — `src/components/LogActivityWidget.tsx:187`

Acceptable workaround for being inside a Sheet/Drawer, but project should standardize z-layer scale (Sheet=50, popover-over-sheet=60, toast=100) instead of one-offs.

## Per-Surface Detail

### Phase 5 — LogActivityWidget (22/24)

**Strengths:** Visual hierarchy is exemplary — the Target icon + UPPERCASE header anchors the widget; the card uses the same `bg-muted/30 border border-border rounded-lg p-3` rhythm as every other section in ProspectSheet. The progressive disclosure of the follow-up sub-form (gated on toggle) is exactly right. Spacing matches the rest of the app (gap-2/gap-3, p-3, space-y-3). The widened touch target on the toggle (`py-1.5 px-2 -mx-2`, line 152) is a recent UX-V2-02 fix and meets the 44px minimum. Submit button is correctly disabled during async work (line 204).

**Weaknesses:**
- The submit copy length-shifts (M6 above)
- The `select` element uses native HTML rather than Radix Select, so styling drifts cross-platform (safari shows native chrome, chrome shows truncated arrow). This is consistent with the rest of ProspectSheet but worth flagging at L-tier.
- The select's `text-xs` (line 126) gives a smaller hit target than the input next to it (which is also `text-xs` but wraps in a 36px-tall control). Symmetry is preserved but mobile thumb-targeting is borderline.
- No `aria-required` or label-for-input — the visible-text anchors rely on `aria-label` (line 123) only on the select; the notes input has none.

### Phase 6 — RecommendationCard (22/24)

**Strengths:** Information density is calibrated well — header chip + 1-3 callouts + 1 sentence + score number is exactly as much as the eye can carry on a slide-over. The severity color mapping (`severityClass`, line 16-20) is principled. The `compact` prop hook (line 36, 41-44, 67-69) means future mobile/inline reuse is already costless. `useMemo([prospect])` (line 36) prevents re-renders.

**Weaknesses:**
- `text-[10px]` and `text-[11px]` proliferate (M1)
- Severity colors are direct `amber-500` / `destructive` mix — `info` and `warn` use direct palette while `critical` correctly uses the `destructive` token. Fix: add `--warning` token usage. (`--warning` is defined in CSS but unused in this file; `bg-warning/15 text-warning` would be cleaner.)
- The `style={{ color: rec.scoreColor }}` inline (line 53) is a token escape (L1).
- No empty-state visual when `callouts.length === 0` — only the suggestedAction sentence renders, which can read like a 1-line throwaway. Consider adding a calm "All clear" affordance for the empty case (current copy: "Maintain cadence — {p.name} is on track for normal follow-up." — works, but a green check icon would help).

### Phase 7 — PipelineForecastBar (18/24)

**Strengths:** The three-tier structure (headline / progress strip / segmented bar + legend) is a textbook financial-data card. The threshold-driven progress color (`bg-emerald` at 100%, `bg-amber` at 70%, `bg-primary` otherwise — line 56-59) aligns with how the rest of the app signals state. Empty state (line 46-53) is well-designed.

**Weaknesses:**
- **a11y on segments is the most serious issue across all five surfaces** (H1) — `aria-label` on `<div>` is dead weight; the segments aren't keyboard-focusable at all. Tooltip access is mouse-only.
- Headline reflow on mobile (M2) — the `ml-auto` on the % Quota column collides with `flex-wrap`.
- 8 distinct color values (slate/blue/indigo/violet/amber/orange/emerald + primary) for stage encoding is one or two more than the eye can categorize. Fine for a static reference card, slightly noisy for at-a-glance reading.
- No "stage X is being closed this month" or close-date awareness — flagged in roadmap, not a Phase 7 deliverable.
- Forecast component imports `forecast.ts` which uses palette colors — appropriate exception since this is categorical data encoding (H3 carve-out).

### Phase 8 — MeetingPrepDialog (17/24)

**Strengths:** The forwardRef pattern (line 38, 80-86) cleanly removes 107 lines of state from ProspectSheet. The six-section markdown contract is forgiving and stable. ReactMarkdown integration is clean. Three actions (Copy / Export PDF / Regenerate) are clearly grouped at the bottom — good Fitts'-law density.

**Weaknesses:**
- **No `DialogDescription`** (H2) — the most important a11y miss in this batch.
- **No `aria-live` on loading region** (H2)
- The PDF export uses inline hex hardcodes that drift from app primary (M4) — the printed brief looks like Tailwind blue-600, not Yext indigo.
- The dialog has `max-h-[85vh]` (line 131) and the inner content has `max-h-[55vh]` (line 146) — these magic numbers are awkward but the only way to get a scrolling content area inside a max-height dialog. Consider switching to flex column with `flex-1 min-h-0` (already partially applied at line 145).
- "None on file." italic placeholder (line 155) is good but has no visual differentiation between "this section was returned empty" vs "the LLM omitted this section" — both produce the same string.
- Copy / Export PDF / Regenerate buttons at line 162-170 wrap below 320px; consider stacking them in a column or hiding "Export PDF" in an overflow menu on mobile.

### Phase 9 — TodayPage / Briefing (18/24)

**Strengths:** The five-section layout (Hero / Inbox-zero / Today's Plan / Overdue Tasks / Going Stale / New Pipeline) reads top-to-bottom with proportional weight. Each row is a single-tap navigation to the prospect — affordance is obvious. The `Stat` helper component (line 254-275) is a tidy abstraction. Inbox-zero celebration (line 76-82) is a delightful touch. The print stylesheet itself is thoughtfully constructed (kills sticky positioning, neutralizes glass backgrounds, removes max-width).

**Weaknesses:**
- **Print stylesheet `button { display: none !important }`** (C1) is overly broad.
- `text-[10px]` on stat labels (M5)
- The "Going Stale" reason string is constructed in `briefing.ts:118` as `"${p.priority} · ${daysStale}d since touch · score ${score}"` — readable but not high-impact. Could lead with the *action* ("Re-engage Hot — 31d quiet") instead of restating the data already shown via the score badge.
- New Pipeline section (line 199-228) uses an emerald accent that isn't tokenized (H3)
- `data-no-print` is applied on the header (line 39) but not on the inbox-zero card (line 77) — when printed during inbox-zero, the user gets a print of "Inbox zero" with nothing else, which is technically correct but reads weird as a printed artifact.
- No print preview affordance — user has to know `Cmd+P` works. A "Print briefing" button (with `data-no-print` parent) would be a 5-min add.

## Recommendations

In priority order:

1. **Fix C1** — change `button { display: none !important; }` to `[data-no-print] button { display: none !important; }` in `src/index.css:365`. Verify by adding a test print-button to TodayPage and confirming print preview hides it via the data-no-print parent, not the global button rule.

2. **Fix H1** — make PipelineForecastBar segments focusable. Either wrap each `<div>` in `<button type="button" className="contents">` (preserves layout) or replace the div with `<button>` directly and use `disabled` styling for non-interactive feel. Either way, the shadcn Tooltip will then activate on keyboard focus.

3. **Fix H2** — add `<DialogDescription className="sr-only">…</DialogDescription>` to MeetingPrepDialog and wrap the loading state in `role="status" aria-live="polite"`. Both are 1-line changes.

4. **Plan H3 migration** — schedule a "tokenize semantic color" task for the next polish pass: introduce `--info` token, route all `amber-*` severity uses through `text-warning bg-warning/15`, all `emerald-*` celebratory uses through `text-success bg-success/15`. Leave `STAGE_BAR_COLORS` alone (categorical exception). Touches ~17 files.

5. **Address M1/M5 in one PR** — find/replace `text-[10px]` → `text-xs` in app code; audit by hand any case where the smaller size is truly required (badge counts in tight nav slots probably need to stay).

6. **Consider M2** — restructure PipelineForecastBar headline as `grid grid-cols-2 sm:flex` so mobile reflows cleanly.

7. Defer M3, M4, M6, all L-tier — track as future polish.
